# Gemma4 한글 Intent 분류 벤치마크 - 실험 기록

## 1. 실험 개요

- **목적**: Gemma4 모델 크기별 한글 Intent 분류 성능 비교
- **도메인**: 고객 서비스 (주문/배송/환불/문의 등 10개 Intent)
- **환경**:
  - Server: Azure VM (Ubuntu, 31GB RAM, CPU only, GPU 없음)
  - Ollama: v0.20.4
  - Public IP: 20.200.215.136

## 2. 대상 모델

## 4. 1차 벤치마크 실행 (실패)

### 4.1 설정

- **API**: `/api/generate` 엔드포인트
- **옵션**: `temperature: 0`, `num_predict: 20`
- **프롬프트**: system prompt + 사용자 메시지를 단일 prompt로 전달

### 4.2 결과

| 모델 | 정확도 | 평균 응답시간 | 비고 |
|------|--------|-------------|------|
| gemma4:e2b (2.3B) | **42.1%** (1263/3000) | 1.3s | 완료, 정확도 낮음 |
| gemma4:latest (4.5B) | **88.8%** (2663/3000) | 1.4s | 완료, 양호 |
| gemma4:26b (MoE) | **0.0%** (0/3000) | 2.6s | ⚠️ 전체 빈 응답 |
| gemma4:31b (Dense) | 측정 불가 (569/3000 중단) | 16.0s | ⚠️ 전체 빈 응답 |

### 4.3 원인 분석: 26b/31b 빈 응답 문제

**증상**: 26b와 31b가 모든 질문에 빈 문자열(`""`)을 반환

**디버깅 과정**:

1. Raw API 응답 확인 → `response: ""`, `done_reason: "length"` 발견
2. `done_reason: "length"`는 `num_predict` 한도 도달을 의미

**근본 원인: Thinking 토큰의 num_predict 예산 소모**

Gemma4 26b/31b 모델은 **thinking 모드가 기본 활성화**되어 있음.
모델이 응답을 생성할 때 내부적으로 "사고 과정" 토큰을 먼저 생성하는데,
이 토큰이 사용자에게는 보이지 않지만 `num_predict` 한도에 포함됨.

```
[사용자 질문]
  → 🧠 thinking 토큰 (숨겨짐, ~200-1000토큰): "사용자가 택배 배송을 묻고 있으니..."
  → 📝 실제 응답 토큰 (보여짐): "배송조회"
```

`num_predict: 20` 설정 시:
- e2b/e4b: thinking 없음 → 20토큰 전부 실제 응답에 사용 ✅
- 26b/31b: thinking에 20토큰 소진 → 실제 응답 토큰 0개 → 빈 응답 ❌

**검증 실험**:

| 설정 | 26b 응답 | 비고 |
|------|---------|------|
| `/api/generate`, num_predict: 20 | `""` (빈 응답) | ❌ thinking이 예산 소모 |
| `/api/generate`, num_predict: 100 | `""` (빈 응답) | ❌ 아직 부족 |
| `/api/chat`, num_predict: 500 | `"배송조회"` | ✅ thinking + 응답 가능 |
| `/api/chat`, think: false, num_predict: 20 | `"배송조회"` | ✅ thinking 비활성화 |

## 5. Thinking 모드 비교 실험

### 5.1 동일 질문으로 thinking ON/OFF 비교 (gemma4:26b)

질문: "사이즈가 안 맞아서 교환하고 싶어요" (정답: 교환요청)

| 설정 | 소요시간 | 응답 | thinking 길이 |
|------|---------|------|--------------|
| think OFF, num_predict: 50 | 26.4s | ✅ 교환요청 | 0자 |
| think ON, num_predict: 500 | 31.5s | ✅ 교환요청 | 875자 |
| think ON, num_predict: 1000 | 17.1s | ✅ 교환요청 | 547자 |

### 5.2 num_predict 상한

`num_predict`의 이론적 최대값은 모델의 context_length:

| 모델 | context_length | num_predict 최대 |
|------|---------------|-----------------|
| e2b / latest | 131,072 (128K) | 131,072 |
| 26b / 31b | 262,144 (256K) | 262,144 |

Intent 분류 작업에서의 실용적 권장값:
- **think OFF**: `num_predict: 50` (충분)
- **think ON**: `num_predict: 500~1000` (thinking + 응답 여유)
